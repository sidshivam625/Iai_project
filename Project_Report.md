# Prompt Injection Detection System: Training & Evaluation Report

## 1. Project Overview & Architecture
This project implements a multi-layer ensemble defense system designed to protect Large Language Models (LLMs) from adversarial prompt injections, jailbreaks, and malicious instructions. Rather than relying on a single point of failure, the architecture utilizes a "Defense in Depth" approach spanning multiple models and heuristics:

*   **Layer 1 (Deep Semantic Representation):** A pre-trained **RoBERTa-base** model fine-tuned to classify prompts natively and extract high-dimensional semantic embeddings (768 dimensions per prompt).
*   **Layer 2 (Statistical Machine Learning):** An **XGBoost Classifier** that trains on the contextual embeddings generated by RoBERTa to detect subtle non-linear patterns.
*   **Layer 3 (Decision Fusion):** A dynamic weighted fusion algorithm combining probabilities from RoBERTa and XGBoost to calculate a unified risk score.
*   **Layer 4 (Behavioral Anomaly Detection):** An **Isolation Forest** trained purely on benign data embeddings to flag syntactically novel or out-of-distribution prompts.
*   **Layer 5 (Knowledge/Rule Engine):** A heuristic rule engine checking for known adversarial strings and patterns.
*   **Layer 6 (Final Risk Aggregator):** The final component that calculates a weighted risk score spanning `[0.0, 1.0]` across all layers and uncertainties to issue a final `BLOCK`, `SANITIZE`, or `ALLOW` decision.

## 2. Data Preprocessing & Pipeline Strategy

A robust, diverse dataset is critical for preventing the model from over-indexing on specific jailbreak formats. The data was aggregated from multiple sources (`Benign data` and `data` directories).

### Preprocessing Steps
1.  **Normalization:** Text was standardized by removing excess whitespace, carriage returns (`\r`), and newline characters (`\n`).
2.  **Filtration:** Corrupt rows, unnamed columns, and empty texts (`""`) were aggressively dropped.
3.  **Deduplication:** A strict deduplication pass was done to ensure the training model wouldn't artificially inflate its confidence on repeated instances of the same prompt.
4.  **Stratification:** The data was split ensuring the exact ratio of Malicious vs. Benign labels was preserved across all sets.

### Dataset Splits
A massive dataset of **180,338 unique, deduplicated prompts** was compiled and split into standard `70/15/15` ratios:
*   **Training Set:** `126,236` samples
*   **Validation Set:** `27,051` samples
*   **Test Set:** `27,051` samples *(20,890 Benign, 6,161 Malicious)*

## 3. Kaggle Training Environment & Parameters

Due to the massive scale of the dataset (126,000+ training strings), local CPU training was financially unviable and extraordinarily slow. The training pipeline was zipped, uploaded to Kaggle, and run using Kaggle's free **Nvidia Tesla T4x2 GPUs** (giving up to 15GB/16GB of VRAM). 

By optimizing the data loaders to securely pin memory (`pin_memory=True`) and modifying the `PromptDataset` to pre-tokenize all 126k inputs in one massive chunk before the PyTorch training loop initialized, the CPU bottleneck was bypassed. This allowed the GPUs to ingest data at maximum bandwidth, cutting training time from several days down to just **~15 minutes**!

### Core Training Parameters
**RoBERTa Semantic Model**
*   **Epochs:** 2
*   **Batch Size:** 64
*   **Max Sequence Length:** 256 tokens
*   **Learning Rate:** `2e-05` (AdamW Optimizer)
*   **Device Context:** CUDA Backend

**XGBoost Classifier (ML Layer)**
*   **Estimators (Trees):** 300
*   **Max Depth:** 6
*   **Learning Rate:** 0.08
*   **Subsample / Colsample:** 0.9 / 0.9

## 4. Evaluation Results & Metrics

The test set containing `27,051` unseen prompts was evaluated locally using the trained models downloaded from Kaggle. The results show extraordinary precision across the board.

### Key Performance Identifiers (Best Standalone Model: XGBoost on Embeddings)
*   **Overall Accuracy:** `99.39%` 
*   **F1-Score:** `0.9866`
*   **Precision (Malicious):** `99.02%` (Out of every 100 prompts it blocks, 99 were legitimately attacks, resulting in near-zero false positives).
*   **Recall (Malicious):** `98.30%` (It successfully caught 98.3% of all possible attacks in the dataset).

### Classification Report Comparisons

| Agent Layer | Benign F1 | Malicious Precision | Malicious Recall | Malicious F1 | Overall Accuracy |
| :--- | :---: | :---: | :---: | :---: | :---: |
| **1. RoBERTa (Raw)** | 0.9955 | 0.9931 | 0.9760 | **0.9844** | 99.30% |
| **2. XGBoost (ML)** | 0.9961 | 0.9902 | 0.9830 | **0.9866** | 99.39% |
| **3. Fusion Ensemble**| 0.9961 | 0.9902 | 0.9830 | **0.9866** | 99.39% |
| **Final Risk Engine**| 0.9959 | 0.9905 | 0.9817 | **0.9861** | 99.37% |

> [!NOTE]
> While the standalone XGBoost scored microscopically higher in isolated testing, the **Final Risk Engine** aggregates uncertainty and anomaly scores, making it mathematically safer against "zero-day" attacks or entirely new jargon that bypasses the standard distributions. 

### Decision Engine Thresholds (Post-Calibration)
The Fusion orchestrator automatically calculated the optimal blocking limits based on the Validation set distribution:
*   **`t_block` (Hard Block Limit):** `0.50`
*   **`t_sanitize` (Warning Limit):** `0.20`
*   Final Weights: `alpha = 0.5`, `beta = 0.2`, `gamma = 0.3`, `delta = 0.1`
